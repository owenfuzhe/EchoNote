# ADR-002: 区分文章与笔记

## 状态

已采纳

## 决策时间

2026-03-17

## 背景

EchoNote 的产品方向不是单一的笔记应用，而是融合了：

- Cubox 的收藏与阅读
- Notion / Craft 的编辑与整理
- ThetaWave / NotebookLM 的 AI 理解与结构化输出

因此，系统里的内容并不都是同一种东西。

如果把所有内容都当成“同一种可编辑文档”，会出现几个问题：

1. 抓取来的文章被误当成可自由编辑的笔记
2. 用户分不清“原文”和“自己的思考”
3. 后续标注、摘录、转写、AI 派生内容会混在一起
4. 编辑器会变得既不像阅读器，也不像真正的笔记器

## 最终结论

EchoNote 从产品模型上区分两类核心内容：

1. `文章（Article）`
2. `笔记（Note）`

## 定义

### 文章

文章是外部内容或半结构化导入内容，例如：

- 网页抓取
- 微信公众号文章
- 小红书内容
- PDF / 文件导入后的原始材料
- 音频 / 图片 / 网页解析后形成的“来源内容”

文章的目标是：

- 阅读
- 标注
- 高亮
- 摘录
- 结构化理解

文章默认不应该像普通笔记那样被直接重写。

### 笔记

笔记是用户自己的内容，例如：

- 手动新建的笔记
- 语音转写后的个人记录
- 基于文章摘录后形成的总结
- AI 生成并被用户接纳的整理稿

笔记的目标是：

- 编辑
- 重写
- 组织
- 延展
- 形成个人知识资产

## 产品行为原则

### 文章页

文章页应更像“阅读器 + 批注器”，而不是纯编辑器。

核心动作：

- 阅读原文
- 查看快照
- 高亮
- 加批注
- 摘录到笔记
- AI 总结 / 提问

不建议把文章正文直接作为一个大输入框开放编辑。

### 笔记页

笔记页应更像“编辑器 + 工作台”。

核心动作：

- 编辑标题与正文
- 插入 checklist / quote / 分段结构
- 标签管理
- 来源关联
- AI 改写 / 提炼 / 扩写

## 对当前项目的影响

当前的 `DocumentView` 同时承担了：

- 阅读页
- 编辑页
- AI 浮层
- 来源页
- 快照页

这会让页面职责过重，也会导致“文章”和“笔记”的体验互相拖累。

后续应该拆成两条体验：

1. 文章详情页
2. 笔记编辑页

## 第一阶段落地建议

在不大改数据结构的前提下，先这样落地：

1. `type = link / file / image` 的内容，优先按“文章”对待
2. `type = text / voice / ai` 的内容，优先按“笔记”对待

这是过渡规则，不是最终真理。

后续如果需要，可以再增加更明确的字段，例如：

- `content_kind = article | note`
- `derived_from_note_id`
- `source_note_id`

## 为什么这个决策重要

它会直接影响：

- 详情页信息架构
- 编辑器能力边界
- AI 的介入方式
- 数据模型设计
- 后续同步和搜索策略

## 下一步

基于这个决策继续推进：

1. 重构笔记编辑页
2. 补文章详情页的阅读 / 标注方向
3. 让 AI 在两类页面里的角色不同
