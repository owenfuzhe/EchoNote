# EchoNote 编辑体验重构路线

## 更新时间

2026-03-17

## 目标

把当前“能打开内容”升级成真正可用的两条核心链路：

1. 新建笔记 -> 编辑笔记 -> 回看笔记
2. 收藏文章 -> 阅读文章 -> 标注 / 摘录 -> 形成笔记

## 现状问题

当前问题集中在：

1. 新建笔记入口刚刚可发现，但底层还是旧逻辑复用
2. 详情页同时承担阅读、编辑、AI、溯源、导出，职责过重
3. 用户自己的笔记和外部文章没有明确区分
4. AI 功能在页面里偏“打断式”，不是“辅助式”

## 阶段规划

### Phase 1：笔记编辑主链路

目标：让“新建一条笔记”变成真实可用能力。

范围：

1. 新建笔记入口打开真正的编辑页
2. 笔记页强化编辑体验
3. 保存状态、云同步状态可见
4. 标签 / 基本属性入口补齐

交付标准：

- 用户可以顺畅新建、编辑、退出、再次打开笔记
- 不依赖猜测就能知道是否已保存、是否已同步

### Phase 2：文章阅读与标注

目标：把文章体验从“可编辑文档”改成“可阅读来源”。

范围：

1. 文章详情页更偏阅读器
2. 支持高亮 / 批注 / 摘录
3. 支持一键生成“我的笔记”
4. AI 基于文章做提炼和问答

交付标准：

- 用户清楚知道自己在看“来源”
- 用户可以把来源内容转成自己的笔记

### Phase 3：账号与同步完善

目标：让云端使用真正稳定。

范围：

1. 匿名用户升级正式账号
2. 云同步状态更清晰
3. 基础冲突处理与错误提示

### Phase 4：AI 与服务端能力产品化

目标：让 AI 和抓取从 demo 能力变成真实可用能力。

范围：

1. AI 服务端代理
2. 链接抓取服务部署
3. OCR / 转写任务化

## 笔记编辑页的设计方向

目标不是做一个迷你 Notion，而是：

- 有 Craft 那种干净的写作体验
- 有 Notion 那种渐进增强的结构化能力
- 有 ThetaWave / NotebookLM 那种 AI 辅助整理能力

### 当前落地原则

这次先借 Notion 在手机上的三个有效模式，但不做像素级复刻：

1. 顶部动作收敛
2. 标题和正文成为主画布
3. 常用编辑动作放到键盘上方工具条
4. `+` 承接复杂插入，不把所有能力都堆在第一层
5. AI 做成次级整理面板，不默认压住编辑器

保留 EchoNote 自己更重要的一层产品判断：

1. 文章是来源，默认只读
2. 笔记是用户自己的内容，可以持续编辑
3. AI、溯源、导出属于辅助层，不压住主编辑流

技术约束也一起记下：

- iOS 原生 `InputAccessoryView` 不适合作为正文多行编辑器的统一方案
- 所以笔记页键盘工具条统一改成我们自己控制的悬浮工具条，保证标题和正文行为一致

### 编辑器基座选择

经过对开源方案的筛选，当前建议是：

1. 不 fork 整个笔记应用作为基座
2. 优先引入成熟的 React Native 编辑器内核，再做 EchoNote 自己的 IA
3. 当前最值得做技术验证的方向是 `TenTap`

原因：

1. 它就是为 React Native 富文本编辑做的，方向比 web-only 或 Flutter 方案更贴近我们当前代码栈
2. 它明确面向移动端富文本体验，并参考了 Notion / Google Docs / Dropbox Paper
3. 它天然比我们继续手搓 `TextInput + 自定义工具条` 更接近成熟编辑器的行为模型

不建议直接 fork 的对象：

1. `AppFlowy`：产品与编辑器都很强，但主栈是 Flutter / Rust
2. `Notesnook`：React Native 路线值得参考，但整应用过重，且产品目标不完全一致
3. `Joplin`：更偏 Markdown / offline-first / notebook 体系，不是我们要的编辑体验方向

### 我们的建议形态

#### 顶部

- 返回
- 页面类型标识（笔记 / 文章）
- 保存 / 同步状态
- 更多操作

#### 主体

- 标题
- 正文主编辑区
- 轻量格式能力

#### 底部工具区

- checklist
- quote
- tag
- AI 助手

#### 次级面板

- 来源信息
- 溯源
- 导出
- 快照 / 原网页

## 当前结论

1. 不继续扩大 `TextInput + 自定义工具条` 的复杂度
2. 笔记编辑页优先迁移到成熟编辑器内核上
3. `文章` 仍保持独立阅读/标注链路，不和 `笔记编辑器` 共用一套重编辑模型
4. 在 `TenTap` 上做二次开发时，优先避免遮挡 WebView 触点和重复内容回灌这两类高频集成陷阱
