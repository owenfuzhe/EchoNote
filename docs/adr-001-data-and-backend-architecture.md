# ADR-001: EchoNote 数据与后端架构决策

## 状态

已采纳

## 决策时间

2026-03-17

## 这份文档是干什么的

用来记录一个会影响后续很多迭代的核心判断：

- 笔记数据到底怎么存
- AI 和抓取能力应该放在哪
- 后续数据库结构变化时，怎么避免反复推翻重来

这份文档尽量用业务语言写，方便后续回看。

AI 助手的能力分层补充说明见：

- [docs/ai-assistant-architecture.md](/Users/bytedance/Echonote/docs/ai-assistant-architecture.md)

## 最终结论

EchoNote 后续采用一套混合架构：

1. `Supabase` 作为核心数据底座
2. `mobile` 直接读写用户自己的基础数据
3. `AI / 抓取 / OCR / 转写` 这类能力放到服务端函数或独立服务

换句话说：

- 基础笔记数据：尽量直接走 Supabase
- 高风险、高权限、需要密钥的能力：不放在前端

## 为什么这么定

### 1. EchoNote 的核心不是“抓取服务”，而是“用户自己的笔记系统”

这个产品最核心的数据是：

- 笔记
- 标签
- 待办
- 收藏
- 附件

这些都属于典型的结构化用户数据，最适合放在数据库里，并且天然需要：

- 登录
- 同步
- 权限隔离
- 多端一致性

Supabase 在这部分已经提供了比较完整的能力：

- Postgres
- Auth
- Storage
- Row Level Security
- migrations

### 2. AI 和抓取不适合放在前端

以下能力不应该直接由客户端持有密钥或独立调用：

- AI 模型调用
- 复杂网页抓取
- OCR
- 音频转写编排
- 限流和计费控制

原因：

- 需要密钥
- 容易超时
- 容易失败
- 存在滥用风险
- 后期通常需要审计和监控

所以这类能力应放在服务端。

### 3. 不建议为了基础 CRUD 再养一层“传统后端”

如果笔记的增删改查也全部走自建 Express API，会带来额外成本：

- 多维护一层接口
- 权限控制更重
- 迭代速度更慢
- 后期移动端和后端容易重复改动

所以更适合的方式是：

- 基础数据直接走 Supabase
- 特殊能力再通过服务端补充

## 我们后续推荐的系统边界

### A. Mobile 端负责

- 用户登录态
- 笔记、标签、待办、收藏的基础读写
- 本地缓存和离线兜底
- UI 交互和用户体验

### B. Supabase 负责

- 用户认证
- 核心业务表
- 文件存储
- 权限隔离
- schema 演进

### C. 服务端函数 / 独立服务负责

- AI 对话和 AI 提炼
- AI action / job 编排
- AI artifact 生成与状态管理
- 第三方网页抓取
- OCR
- 转写增强
- tools / MCP 等外部能力接入
- 需要高权限或密钥的后台逻辑

这里的关键点不是“所有 AI 都放到一个黑盒后端里”，而是：

- 同步能力通过 `action` 暴露
- 重型生成通过 `job` 承担
- 最终产物沉淀为 `artifact`
- 供应商和外部工具都留在服务端抽象层

## 这对当前项目意味着什么

### 现在先做什么

优先级最高的是：

1. 在 Supabase 建立第一版正式 schema
2. 把 notes / tags / todos 这些核心数据放进去
3. 让 mobile 开始连 Supabase
4. 保留本地缓存作为兜底

### 现在不急着做什么

以下内容不该作为当前第一步：

- 把所有能力都先塞进一个自建 Node backend
- 先追求完整 AI agent
- 先做复杂抓取编排
- 先做很重的运维体系
- 先做完整 MCP / 插件平台

## 关于“数据库以后会变，怎么办”

这是正常情况，不是坏事。

正确做法不是“尽量别改表”，而是：

1. 从现在开始使用 migrations
2. 优先做可兼容的增量变更
3. 尽量让前端只感知稳定的数据结构
4. 大改时分阶段迁移，不直接硬切

## 第一版建议的数据模型

为了让 mobile 端可以直接连数据库、减少多表事务和复杂后端编排，第一版建议先采用更贴近当前产品结构的模型：

- `profiles`
- `notes`

其中：

- `tags` 先放在 `notes.tags`
- `todos` 先放在 `notes.todos_json`

后续如果任务系统、标签系统变复杂，再拆成独立表：

- `tags`
- `note_tags`
- `todos`
- `attachments`
- `capture_jobs`
- `ai_runs`

## 当前阶段的执行原则

后续推进时遵守这几个原则：

1. 先做可上架的基础能力，再做高风险亮点
2. 先把数据链路打通，再追求复杂智能能力
3. 重大技术选型优先查最佳实践，再落代码
4. 每次关键架构判断都记录到 docs，方便回溯
5. 前端只感知稳定的能力接口，不直接绑定某个 provider 或 tool

## 当前对数据库和登录态的补充判断

基于当前项目推进情况，再补一条更具体的判断：

1. `notes` 的核心 schema 现在不需要推翻重做
2. 当前更值得补的是“结果归属”，尤其是 AI jobs / artifacts 的 `ownerId`
3. 登录态先保持“匿名可用”，正式账号放在 MVP 主链路稳定之后

也就是说：

- 第一版核心数据结构继续以 `profiles + notes` 为主
- 不急着把 tags / todos / artifacts 全部拆成复杂多表
- 但所有需要沉淀的 AI 结果，都应该尽早带上用户归属维度

这样做的好处是：

- 不会为了追求完美 schema 拖慢产品验证
- 后续做 `Sign in with Apple` 时，不需要再回头给 AI 数据补归属关系
- 可以保持“先匿名试用，后升级正式账号”的产品节奏
