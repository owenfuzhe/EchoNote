从 PRD（产品需求文档）的视角来看，“Context Graph（上下文图谱/决策痕迹）”的核心是将 EchoNote 从一个“存放结果的静态仓库”变成一个**“记录思考过程的动态引擎”**。

为了让开发团队和设计师能看懂并落地，我们需要把刚才的宏观概念拆解为具体的**功能模块 (Features)** 和 **用户故事 (User Stories)**。

以下是为 EchoNote 深度定制的 **Context Graph 核心模块 PRD 节选**：

---

### 模块一：笔记溯源面板 (Context Trace Panel)

**对应概念：** 记录“Decision Traces（决策痕迹）”，解决“为什么会有这条笔记/方案”的问题。

* **用户故事 (User Story)：**
作为一名知识工作者，当我回看一个月前由 AI 帮我生成的一份《竞品调研报告》时，我希望知道这份报告是基于哪些原始碎片（截图、语音、链接）和哪些指令生成的，以便我评估其时效性和准确性。
* **功能描述 (Functional Requirements)：**
1. **隐式记录**：每当系统（通过 DeerFlow 或本地大模型）生成一段总结、方案或 Podcast 播客时，后端必须生成一个 JSON 对象，绑定在此笔记实体上，记录：
* `Trigger`: 是系统定时触发，还是用户手动长按 ♾️ 触发？
* `Inputs`: 引用的所有源文件 ID（如：语音备忘录 #12, 网页剪藏 #45）。
* `Prompt/Overrides`: 用户在生成过程中手动介入的修改指令（如：“只保留商业落地部分”）。


2. **显性展示**：在单篇笔记的底部，增加一个折叠的 **“溯源 (Trace)”** 区域。


* **UI 交互草图：**
```text
+------------------------------------------+
|  📄 竞品调研报告 (最终态)                 |
|  ... (正文内容) ...                       |
|                                          |
|  --- 🔽 展开溯源 (Context Trace) -------- |
|  [ 🧠 生成于 2026-03-01 14:00 ]           |
|  基于以下 3 个上下文缝合：                |
|  1. 🔗 [网页剪藏] A公司定价策略            |
|  2. 🎙️ [语音碎片] "重点看A公司的企业版"     |
|  3. 💬 [用户干预] "帮我删掉技术架构对比"    |
|                                          |
|  [ ⏪ 重放生成过程 ]  [ 🔄 添加新信息重写 ] |
+------------------------------------------+

```



---

### 模块二：动态先例引擎 (Dynamic Precedent Engine)

**对应概念：** 记录用户的“例外与偏好 (Exceptions & Precedent)”，让 AI 越用越顺手。

* **用户故事：**
作为一个有特定工作习惯的用户，当我多次手动修改 AI 生成的格式（比如我总是把播客主持人改成“理性男声”，或者总是把要点总结改成 Markdown 表格），我希望系统能自动记住这个“先例”，下次直接应用，而不需要我每次都重新设置提示词。
* **功能描述：**
1. **行为捕获 (Action Hook)**：监听用户在编辑区的行为。如果用户对 AI 生成的内容进行了 >30% 的结构性删改，或者连续 3 次更改了默认配置（如播客音色）。
2. **图谱更新 (Graph Update)**：AI 在后台静默提取这个“变更意图”，并写入该用户的 Context Graph 偏好节点中。
3. **用户感知 (Toast Notification)**：系统不需要复杂的设置页面，而是通过轻提示告知用户。


* **UI 交互逻辑：**
* *用户操作*：用户连续第二次在“Podcast Remix”时手动去掉了“背景音乐”。
* *系统反馈*：界面顶部弹出一个轻量级的紫色 Toast：
* ✨ *“已将您的先例记入图谱：未来的科技类播客将默认关闭背景音乐。 [撤销]”*





---

### 模块三：跨系统意图捕获 (Cross-Source Synthesis)

**对应概念：** 打破信息孤岛，完成跨系统的上下文综合。

* **用户故事：**
当我随手拍了一张发票，接着又长按右下角 ♾️ 录了一段语音说“这是昨天请客户吃饭的钱”，我希望系统能自动把这两件“不在同一时间、同一格式”的事情缝合在一起，生成一条完整的财务记录。
* **功能描述：**
1. **时间窗聚合 (Time-window Clustering)**：当用户通过右下角 Pincer Hub 输入碎片时，如果两次输入间隔在 10 分钟内，且语义（Vector 距离）相近，系统自动建立“上下文强连接 (Strong Edge)”。
2. **实体融合 (Entity Resolution)**：后台 Agent 提取图片中的金额，并将其与语音中的“请客户吃饭”意图合并。


* **UI 交互草图 (首页动态流展示)：**
```text
+------------------------------------------+
|  ✨ AI 智能融合 (Synthesis)                |
|  已将您刚上传的 [📸 图片: 餐饮发票 $120]     |
|  与 5 分钟前的 [🎙️ 语音: 招待 A 公司张总]    |
|  合并为一条完整记录。                       |
|                                          |
|  [ 📂 归档至: 报销记忆集 ] [ 🚀 发送给财务 ]|
+------------------------------------------+

```

### 模块四： Explore Tab 中的“决策回放 (Idea Time-Machine)”

**对应概念：** 将 Context Graph 具象化为可视化的商业资产。

* **用户故事：**
当我构思一部小说或者一个大项目时，我希望能在 Explore（探索）Tab 里，不仅看到网状的节点，还能像看视频一样，按时间轴回放一个伟大的想法是如何从几个碎单词，经过无数次搜集和 AI 迭代，长成参天大树的。
* **功能描述：**
1. 在 Explore Tab 的“知识图谱”视图中，增加一个 **“Timeline (时间轴)”** 滑块。
2. 拖动滑块时，图谱上的节点会按照创建和被引用的时间顺序，依次亮起和连线。



---

### 💡 PRD 落地评估总结：

把 Context Graph 引入 EchoNote，本质上是增加了**两个数据库表/逻辑层**：

1. **实体表 (Entities)**：存笔记、语音、图片（现在的 EchoNote 已经有了）。
2. **痕迹表 (Traces/Edges)**：存“谁在什么时间，基于什么前置条件，执行了什么动作，产生了什么结果”（这是形成护城河的关键）。
